Dynamic Adaptation Using Model (DAUM) - INRIA/IRISA, France

This project has been created in the case of collaboration with a department of firefighters of Brittany.

The goal is to build a real-life, multi-user, real time decision system for crisis management that are able to be adapt according to environmental information.

Required Features

  • computation nodes are Android-based tablets with limited computing resources, used by field personnel;
  • passive nodes are sensors embedded in personal protective equipment, with very low computing capabilities (e.g. temperature sensors);
  • wireless networks connect these nodes; the technologies that we use are ad-hoc Wifi mesh networks, Xbee R mesh networks, and GSM cellular networks;
  • the applications running on the interconnected computation nodes are real-time, multi-user cooperative data editors working peer-to-peer;
  • the business domain of the application includes life critical procedures that must be performed in real time.

At the application level, each personal tablet maintains a full business model of the situation (position and type of hazard sources and targets, position and tactical missions of emergency personnel and vehicles, position and status of resources such as water lines, etc.). With a personal tablet each officer can update tactical information, make decisions and share them with other officers’ tablets, with the main command post and even with tablets of en-route officers. The main components of the application are a navigation system (maps and GPS), several resource management systems (engine position, assignment, reinforcement, etc), a tactical situation editor (SITAC) and a data bridge connected to the main command center computer system. Android tablets and computers of the command center can host components powerful enough to actively participate in the distributed computation of Kevoree’s models at runtime. Sensors in personal protective equipments are passive nodes that can share a Kevoree model view of their current dynamic configuration.

Resource driven versus activity driven adaptation

This system must adapt itself to changes in the available resource set, for instance when a better network link can be established between the tablets, using a field wireless hotspot. This kind of adaptation will also occur when more and more resources are added because of reinforcements arriving on site. Another important adaptation situation comes from changing user needs. For instance, an escalation of response implies tactical decisions that include a significant reconfiguration of the team (e.g. scaling up or scaling down the organization, reassigning responsibilities, etc). To save resources (e.g. memory, processor and network bandwidth), nodes are regrouped automatically to match the allocation of personnel on the different sectors on the field. For instance, at some point a decision is taken to setup a field hospital 1 km away. Our system must then anticipate this possible evolution both on the functional side (assist command in reallocating personnel and vehicle, setting up missions for group leaders using predefined patterns, etc) and on the system side (reallocating node functions, changing link configurations, setting up bridges with “foreign” systems). To anticipate this kind of reconfiguration, our system must maintain a current configuration (at business level and at platform level) and it must also prepare future configurations, using “what if” scenarios. In some way, the whole system aims at managing the resources (computation and communication) optimally, and to this aim it makes use of predictive models to anticipate changes, evaluate new architectures, prepare them and activate them.

Challenges

The DAUM system must manage several problem dimensions.

  • Embedded sensor networks

    Many sensors (e.g. body temperature sensors embedded in clothes) cannot fully participate in the distributed models at runtime algorithms, mainly because of hardware limitations. In this use case the uncoupled reflection model allows for interacting and reconfiguring small devices asynchronously. For example, a limited computational node can easily host a model representation of its firmware (only a few Kbytes) but does not have the capability to dynamically reconfigure itself. In this case, the intelligent reflection layer allows larger nodes (i.e. tablet or coordinator computer) to seamlessly reason about these sensors using their internal representation, without requiring a call to the reflection layer on these limited devices.

  • Pervasive environment

    In a pervasive context, the management of communication issues and its associated compensation policies are critical tasks. These tasks are even more critical in a sporadic context where peers can appear and disappear without a warning. Typically, tablets and even more sensors are disseminated on a field of action without a proper stable network infrastructure, with propagation problems caused by buildings or landscape. As we rely on dynamic adaptation of the system architecture using models at runtime, we must design distributed algorithms that preserve consistency of the distributed data and provide adequate consensus to enforce consistent architecture reconfigurations on the fly.
